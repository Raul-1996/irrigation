# WB-Irrigation — Аудит коммерческой готовности (2026-05-09)

**Цель**: подготовить продукт к модели «1 клиент = 1 инстанс» (single-tenant).
**Режим**: read-only, без правок и деплоя.
**Pipeline**: 4 параллельных профильных архитектора (Security / Backend / Frontend / DevOps) → консолидация → Stage 2 фиксы.
**Ветка**: `audit/commercial-readiness`.

## TL;DR

Проект **технически работающий**, но к коммерческой single-tenant продаже **не готов**. Главные блокеры:

1. **Хардкод кредов в Cloudflare Worker** (`Poliv:Poliv` — одинаков для всех клиентов). Это **первый и главный блокер** — без устранения нельзя продавать ни одному внешнему клиенту.
2. **Дефолтный пароль `1234`** в миграциях, без принудительной смены при первом входе.
3. **MQTT-креды и initial-password в plaintext**: в БД хранятся открыто, в логах попадают (PII-фильтр их не ловит), на диске world-readable.
4. **Нет идентификатора клиента** (`controller_id` / `customer_id`) — невозможно различить инстансы для биллинга, поддержки, мониторинга, обновлений.
5. **Отсутствует процесс безопасного апгрейда + бэкап-стратегия**: `update_server.sh` не атомарен, нет down-миграций, бэкапы только локально на eMMC контроллера.
6. **Параллельные реализации шедулера** (`irrigation_scheduler.py` 1649 LOC vs новый `scheduler/` package 1337 LOC) — drift, в проде используется монолит, тесты только на новый.
7. **Frontend перформанс деградации**: `Math.random()` в классификаторе раз в секунду (мерцание UI), `window.fetch` обёрнут 4 раза подряд, фуллпейдж innerHTML re-render каждые 5с.
8. **Физическая безопасность**: SIGKILL процесса оставляет клапаны открытыми → потенциальный пролив.

**Сводно**:
- **18 P0** (критично, блокирует коммерческое использование)
- **29 P1** (важно, исправить до выхода на платных клиентов)
- **42 P2** (полировка, можно итеративно после релиза)

## P0 — блокеры коммерческой готовности

### 1. Аутентификация / Multi-tenant identity

| ID | Источник | Описание |
|----|----------|----------|
| **P0-01** | Security NEW-04, DevOps P0 #2 | **CF Worker hardcoded `Poliv:Poliv`** — одинаков для всех клиентов, в репозитории как plaintext в `wrangler.toml`. Утечка одного логина = доступ ко всем будущим инстансам. |
| **P0-02** | DevOps P0 #1, Security NEW (initial password) | **Дефолтный admin/`1234`** в `db/migrations.py:195,208` без `must_change_password=True`. Установится у каждого клиента и будет жить до первой ручной смены. |
| **P0-03** | DevOps P0 #3 | **Нет `controller_id` / `customer_id`** — нельзя адресовать инстанс из CF, нельзя писать аналитику/биллинг/инциденты per-customer. |
| **P0-04** | Security P1 (HMAC secret) + Worker hardcode | HMAC-secret для cookie тоже хардкод в Worker. Один клиент скомпрометирован = подделать cookie у всех. |

### 2. Утечка секретов / физическая безопасность данных

| ID | Источник | Описание |
|----|----------|----------|
| **P0-05** | Security NEW-01 | **MQTT-пароль в БД хранится plaintext** в `mqtt_credentials.password` (несмотря на наличие `utils.encrypt_secret`). Кто получил `irrigation.db` — получил MQTT broker. |
| **P0-06** | Security NEW-02 | **`irrigation.db` world-readable** + смонтирован bind-mount, на хосте читается любым процессом / пользователем. |
| **P0-07** | Security NEW-03 | **`/api/env` доступен для guest** без аутентификации, отдаёт кусок конфига. |
| **P0-08** | Security (initial password в логах) | Initial password при `db init` пишется в логи через обычный `logger.info`, **PII-фильтр его не маскирует** → логи становятся секретом. |

### 3. Backend — целостность процесса полива

| ID | Источник | Описание |
|----|----------|----------|
| **P0-09** | Backend P0 #1 | **SIGKILL → клапаны остаются открытыми**. Нет фейл-сейф закрытия при аварийной остановке процесса. Реальный риск пролива. |
| **P0-10** | Backend P0 #2 | **Параллельные реализации шедулера** (`irrigation_scheduler.py` монолит используется в проде, новый `scheduler/` пакет покрыт тестами но не подключён). Drift, баги фиксятся в одном месте, не в другом. |
| **P0-11** | Backend P0 #3 | `recover_missed_runs()` корректно работает только для cron-расписаний по weekday, для interval/datetime пропускает события при перезапуске. |
| **P0-12** | Backend P0 #4 | Закрытие master-крана через `threading.Timer` — не персистентно. Перезапуск процесса = таймер потерян, master не закроется. |

### 4. DevOps — релиз-цикл

| ID | Источник | Описание |
|----|----------|----------|
| **P0-13** | DevOps P0 #4 | `update_server.sh` **не атомарен**, нет `_down_*` миграций, нет rollback. Сломанный апдейт = клиент без воды/полива до ручного фикса. |
| **P0-14** | DevOps P0 #5 | **Бэкапы только локально** на eMMC контроллера. Сгорела SD/eMMC = всё потеряно. Нет offsite, нет restore-теста. |

### 5. Frontend — UX/корректность

| ID | Источник | Описание |
|----|----------|----------|
| **P0-15** | Frontend P0 #1 | **`Math.random()` в классификаторе flow-active** — UI мерцает (зона визуально включается/выключается каждую секунду без реальной причины). |
| **P0-16** | Frontend P0 #2 | **`window.fetch` обёрнут 4 раза** подряд — каждый запрос проходит через 4 уровня перехватчиков, perf-death и непредсказуемая обработка ошибок. |
| **P0-17** | Frontend P0 #3 | **SW не делает `controllerchange` reload** — обновление SW залипает, юзер видит старую версию до явного hard reload. |
| **P0-18** | Frontend P0 #4 | **Нет `confirm()` на старт зоны** — случайный тап в мобильном UI запускает реальный полив. |
| **P0-19** | Frontend P0 #5 | **Глобальный error handler показывает raw stack trace** — утечка путей/переменных конечному пользователю. |

## P1 — важно для платных клиентов (29 пунктов)

### Security (8)
- HMAC-secret хардкод в коде (наряду с CF Worker creds → P0-04 + P1).
- Нет anti-bruteforce (rate-limit на login).
- Initial password в логах (см. P0-08, тут «системно: PII-filter пропускает»).
- Bulk update SQL — динамические имена колонок без allow-list.
- `SESSION_COOKIE_SECURE` default `False`.
- Telegram chat_id используется как auth-id без HMAC.
- CSP `unsafe-inline` для скриптов и стилей.
- Нет `Strict-Transport-Security` / `Referrer-Policy` (хедеры выставляет CF, но Flask должен дублировать на случай LAN-bypass).

### Backend (6)
- Unbounded lock dicts (`zone_locks`, `group_locks`) растут без вытеснения.
- Ephemeral MQTT clients создаются на каждое сообщение, нет переиспользования.
- Нет Alembic — миграции самописные, без версионирования и `_down_`.
- Нет connection pool для SQLite, каждый запрос открывает соединение.
- «Eternal backward-compat stubs» — старые функции, помеченные `# TODO remove in v2`, но v2 не наступит.
- `dlog()` делает SELECT в БД на каждый вызов, для проверки уровня логирования.

### Frontend (8)
- Полный `innerHTML` re-render групп каждые 5 сек (вместо diff-апдейта).
- `updateStatusDisplay` / `refreshSingleGroup` — 150 LOC дублируют друг друга.
- `setAttribute('onclick', ...)` — race с CSP, иногда падает.
- Polling без `AbortController` и без exponential backoff.
- PWA кеширует только иконки, не оффлайн-готов.
- Touch targets <44px (нарушает iOS HIG, мобильная UX страдает).
- `cache_buster` хардкоден `'v'` (а должен быть commit hash или версия).
- `envProbeTimer` без cleanup → утечка таймеров после ухода со страницы.

### DevOps (7)
- Нет multi-controller automation (Ansible / fleet-tooling). Деплой — ручной по SSH.
- Watchdog в двух местах (`sd_notify` + cron) — конфликт, double-restart.
- Нет retention для `logs` и `water_usage` — таблицы растут вечно.
- `/metrics` без auth и без агрегации (per-controller scrape, но нет federation).
- `mosquitto.conf` расходится между Docker-окружением и нативной установкой на контроллере.
- GitHub Actions использует password-SSH (а не key-based) для деплоя.
- LAN bypass: `run.py:69` биндится на `0.0.0.0` → внутри сети auth можно обойти, минуя CF Worker.

## P2 — полировка (42 пункта)

Подробно — в доменных файлах. Среди значимых:

- **Security P2 (8)**: XOR fallback, dead `.env.example`, нет SBOM/CVE сканирования, нет idle-session timeout, `MAX_CONTENT_LENGTH=10MB` (DoS-вектор), audit log recursion, `/logout` GET (CSRF), `mosquitto.conf` без TLS.
- **Backend P2 (10)**: `sse_hub` module-state, `float_monitor` с `sqlite3` fallback, 1s exclusivity poll, дублирующие auth-хуки, `if TESTING:` ветки в проде, jobs.db без backup-дисциплины, ручной `csrf.exempt`, дубль логгеров, дубль кода program/group, Flask dev fallback в `run.py`.
- **Frontend P2 (13)**: a11y (контраст, focus-visible), dead code, проблемы тёмной темы, мелкие виз. артефакты — детально в frontend.md.
- **DevOps P2 (11)**: документация в хаосе, default `SECRET_KEY` fallback, dev-скрипты в корне, `.bat` файлы (Windows-наследие), test results закоммичены, `db/*.db` в репо, пустой `migrations/versions/`, Docker без watchdog, restore не тестировался, нет процедуры SSH-key rotation.

## Stage 2 — план фиксов

Каждый P0/P1 = **отдельный коммит** в эту же ветку. По каждому: Архитектор → Синьор → Критик → Перепроверка → Push. **Деплоя не будет** до явного подтверждения Рауля.

### Очерёдность (рекомендуемая)
1. **P0-01 → P0-04** (auth + multi-tenant): без этого нельзя продавать. Перед началом — обсудить дизайн `controller_id`, формат CF-конфига per-tenant, ротацию HMAC-secret.
2. **P0-05 → P0-08** (секреты в БД/логах/файлах): шифрование MQTT, права на `irrigation.db`, закрыть `/api/env` для guest, починить PII-фильтр.
3. **P0-09 → P0-12** (целостность шедулера): фейл-сейф закрытие клапанов, унифицировать scheduler (одна реализация), persistent timer для master, recover_missed для всех типов триггеров.
4. **P0-13, P0-14** (релиз/бэкапы): атомарный update + down-миграции, offsite backup + restore drill.
5. **P0-15 → P0-19** (frontend критика): убрать random, развернуть fetch-обёртки, controllerchange reload, confirm на запуск, безопасный error handler.
6. **P1**: по доменам, серией коммитов.
7. **P2**: после первого платного клиента, итеративно.

### Чек-лист коммерческой готовности (DevOps P0/P1 объединено)
- [ ] Per-tenant CF Worker creds + ротация HMAC.
- [ ] Forced password change on first login.
- [ ] `controller_id` через всю цепочку (CF, бекенд, метрики, бэкапы).
- [ ] MQTT-пароли только зашифрованы.
- [ ] `irrigation.db` — `0600`, owner = service user.
- [ ] Атомарный update + автоматический rollback на failed migration.
- [ ] Бэкап offsite (S3/B2/etc) + ежемесячный restore drill.
- [ ] Один шедулер, один кодпас.
- [ ] Fail-safe close клапанов на любой exit.
- [ ] Healthcheck + paging (бот / email) — пока нет, но в P1.
- [ ] Документация: install / update / rollback / disaster-recovery — три коротких runbook'а.

## Замечания и риски

- Доменные отчёты местами **пересекаются** (CF Worker creds — упоминают Security и DevOps; MQTT plaintext — Security и DevOps; default `1234` — DevOps). В этом документе они дедуплицированы, но при чтении доменных файлов учитывайте.
- В Stage 2 **порядок имеет значение**: правка фронта без поднятого backend-контракта (например, после переезда на `controller_id`) — двойная работа.
- Часть P1 в Backend (Alembic, connection pool) — это **рефакторинг**, не bug-fix. По Karpathy-принципам — отдельные коммиты, обоснование «это нужно для коммерции», не «потому что красиво».
- **Не делать** на этой ветке: ничего, что не входит в один из P0/P1 пунктов отчёта. Любое расширение скоупа — отдельный тикет, отдельная ветка.
