3. Анализ сильных сторон

✓ Безопасность: SECRET_KEY автогенерация

config.py:12-30 — _load_or_generate_secret() генерирует 64-символьный hex-ключ, сохраняет в файл .secret_key с chmod 0o600. Старый хардкод 'wb-irrigation-secret' игнорируется.

✓ MQTT пароли зашифрованы (AES-256-GCM)

utils.py:82-116 — encrypt_secret() / decrypt_secret() с 12-байт nonce и GCM-тегом. Ключ шифрования генерируется в .irrig_secret_key (32 random bytes, 0o600). Миграция encrypt_mqtt_passwords в db/migrations.py автоматически шифрует plaintext пароли.

✓ CSRF включён глобально

config.py:43-45 — WTF_CSRF_ENABLED = True, WTF_CSRF_CHECK_DEFAULT = True. CSRF-токен в мета-теге, автоматический interceptor для fetch/XHR в base.html. Login endpoint корректно исключён.

✓ QoS 2 для команд реле

Все критические MQTT-публикации (zone_control.py, routes/zones_api.py, groups_api.py) используют qos=2, retain=True. Retry с backoff в mqtt_pub.py:130-170 (до 10 попыток connect + 3 retry для wait_for_publish).

✓ Observed State Verification

services/observed_state.py — после каждой публикации ON/OFF создаётся временный MQTT-клиент, который подписывается на топик зоны и ждёт подтверждения. При таймауте — retry publish до 3 раз. При полном провале — инкремент fault_count, Telegram-алерт. Профессиональный подход уровня промышленных SCADA.

✓ Параметризованные SQL-запросы

Все SQL в db/*.py используют ?-параметры. Прямого конкатенации SQL не обнаружено. SQLite WAL mode + PRAGMA foreign_keys=ON, synchronous=NORMAL.

✓ Хеширование паролей (pbkdf2:sha256)

db/settings.py — werkzeug.security.generate_password_hash с pbkdf2:sha256. Минимум 8 символов, blocklist простых паролей, принудительная смена при первом входе.

✓ Модульная архитектура

Рефакторинг app.py (4411→370) и database.py (2359→306) через Repository Pattern. Зависимости SSE Hub инжектируются через init(). Boot-time инициализация вынесена в services/app_init.py.

✓ Comprehensive watchdog system

Три уровня защиты: 1) services/watchdog.py — cap-time (дефолт 240 мин), concurrent zone monitoring; 2) app.py:_enforce_group_exclusive_all_groups — каждую секунду; 3) APScheduler hard-stop jobs. Telegram-алерты при аномалиях.

✓ Приятные детали

  • Server-Timing header для диагностики (app.py:_perf_add_server_timing)
  • Service Worker для offline-кеширования
  • Health panel (клавиша 'h') и Stopwatch panel (клавиша 's') для диагностики
  • Photo upload с auto-rotate EXIF, resize и WebP conversion
  • Bulk API endpoints (next-watering-bulk, import, duration-conflicts-bulk)
  • Boot-sync: все зоны и мастер-клапаны OFF при старте
  • Early-off configurable (0-15 сек) для плавных переходов
  • Dark theme через @media prefers-color-scheme

4. Анализ слабых сторон

🔴 728 catch-all except Exception

Удвоилось с 356 (прошлый аудит). Многие ловят всё подряд и молча проглатывают ошибки через logger.debug. Это маскирует реальные баги и делает отладку невозможной.

Худшие файлы: irrigation_scheduler.py (92), zones_api.py (82), system_api.py (77), monitors.py (48).

Пример из irrigation_scheduler.py:397:

try:
    okv = self.db.update_zone_versioned(...)
except Exception as e:
    logger.debug("Exception in line_397: %s", e)
    okv = False

Проблема: если update_zone_versioned падает из-за corrupted DB, зона запустится с некорректным состоянием.

🔴 Дефолтный пароль '1234' при первом запуске

db/migrations.py:_insert_initial_data — при пустой базе устанавливается generate_password_hash('1234'). Хотя есть blocklist и принудительная смена, пользователь может просто игнорировать предупреждение.

🔴 SSE-подписка на QoS 0

services/sse_hub.py:281 — client.subscribe(t, qos=0). Если MQTT-брокер перегружен, SSE-хаб может пропустить критическое изменение состояния реле. Для read-path это менее критично, но для observed state consistency — плохо.

🔴 Дублирование логики start/stop

Логика запуска зоны дублируется минимум в 4 местах:

  • services/zone_control.py:exclusive_start_zone
  • routes/zones_api.py:api_zone_mqtt_start (1000+ строк!)
  • routes/zones_api.py:start_zone
  • irrigation_scheduler.py:_run_group_sequence

Каждое место имеет свою версию MV-логики, peer-off, scheduling. При баге в одном месте — другие не исправлены.

🟡 Отсутствие rate-limiting на API

services/rate_limiter.py существует (86 строк), но не подключён ни к одному route. Любой клиент может безлимитно дёргать /api/zones/{id}/mqtt/start. На ARM-контроллере (Wirenboard 8) это может привести к DoS.

🟡 Нет тестового покрытия в CI

Есть 12+ тестовых файлов в tools/tests/, но нет CI-пайплайна (GitHub Actions / GitLab CI). Тесты нужно запускать вручную. Нет coverage-отчётов.

🟡 HTTPS не настроен

Приложение слушает HTTP на порту 8080. В локальной сети Wirenboard это приемлемо, но пароли и CSRF-токены передаются в открытом виде. SESSION_COOKIE_SECURE выключен по умолчанию.

🟡 Нет миграций «вниз» (downgrade)

30 миграций — только вверх. Откатить невозможно, кроме как восстановить из бэкапа SQLite.

🟡 time.sleep в потоках планировщика

irrigation_scheduler.py — в _run_program_threaded и _run_group_sequence используется time.sleep(1) в цикле для ожидания. Это блокирует поток APScheduler. Если нужно параллельно 4 группы — нужно 4 потока, каждый с блокирующим sleep.

7. Детальный разбор конкурентов

Hunter Hydrawise — профессиональный стандарт

Зоны: Pro-HC: 6/12/24; HPC: до 32 модульно. Цена: $300-500.

Главное преимущество: ET-based predictive watering с интеграцией The Weather Company. Flow monitoring + electrical diagnostics. Contractor portal для профессионалов.

Чего нет у WB-Irrigation: ET-расчёт, предиктивный полив, multi-site management, cloud analytics, WaterSense сертификация.

Слабости Hydrawise: Зависимость от облака, закрытая экосистема, нет интеграции с Wirenboard/Modbus, высокая цена.

Rachio 3 — лучший UX для потребителя

Зоны: 8 или 16. Цена: $200-250.

Главное: Weather Intelligence Plus (hyperlocal, 36 feet accuracy), HomeKit/Alexa/Google, no subscription.

Чего нет у WB-Irrigation: Smart home integration, hyperlocal weather, native mobile app, leak detection.

Слабости Rachio: Полная зависимость от облака (без интернета не работает), максимум 16 зон, нет Modbus/промышленных протоколов.

OpenSprinkler — ближайший аналог (open source)

Зоны: До 72 с экспандерами. Цена: $140-183.

Главное: Open source, локальный REST API, Zimmerman weather, community-driven.

Сравнение с WB-Irrigation: OpenSprinkler ближе всего по философии, но использует собственное hardware. WB-Irrigation выигрывает интеграцией с Wirenboard (профессиональный DIN-rail контроллер с десятками I/O модулей).

Слабости OS: Proprietary hardware (нет DIN-rail), нет MQTT нативно, слабый UI на мобильных.

RainMachine — лучший offline-контроллер

Зоны: 8/12/16. Цена: $169-350.

Главное: Полная автономность, HD touchscreen, multi-source weather (MetNo, NOAA, Netatmo), HomeKit.

Чего нет у WB-Irrigation: Touchscreen на контроллере, multi-source weather aggregation, HomeKit.

Rain Bird ESP-TM2, Netro Sprite, Orbit B-hyve, Gardena Smart

Rain Bird: Надёжный, до 12 зон, требует LNK2 WiFi-модуль ($140) для smart-функций. Дорого за ограниченные возможности.

Netro Sprite: До 12 зон, plant intelligence для 1000+ видов, EPA WaterSense. Слабая офлайн-поддержка.

Orbit B-hyve: Дешёвый ($90-200), до 16 зон, NOAA weather. Проблемы с Wi-Fi стабильностью по отзывам.

Gardena: До 6 зон, экосистема для европейских садов, требует gateway. Не подходит для промышленного применения.

9. Уникальные преимущества WB-Irrigation

🏆 Чего нет ни у одного коммерческого аналога

1. Промышленное реле через Modbus/MQTT

Wirenboard wb-mr6cv3 — это DIN-rail реле промышленного класса с 24V питанием, защитой от КЗ, и Modbus-RTU/MQTT интерфейсом. Ни один потребительский контроллер не может управлять реле через Modbus.

2. 24 зоны на реальном hardware

4×6-канальных модуля. Hydrawise поддерживает 32, но за $500+. WB-Irrigation даёт 24 зоны на $100 hardware.

3. Полный контроль и open source

Нет облака, нет подписки, нет vendor lock-in. Код можно модифицировать под любые нужды. Единственный аналог — OpenSprinkler, но у него нет Modbus.

4. Экосистема Wirenboard

Через один контроллер WB8 можно управлять не только поливом, но и: освещением, отоплением, вентиляцией, датчиками (температура, влажность почвы, CO2). Ни один irrigation controller так не может.

5. Observed State Verification

Уникальная фича: после каждой команды приложение подписывается на MQTT и ждёт подтверждения от реле. Если реле не ответило — retry + Telegram алерт + fault counter. У коммерческих аналогов — fire-and-forget.

6. Telegram-бот вместо app

Не нужно устанавливать приложение. Управление через Telegram — мессенджер, который уже у всех. Inline-кнопки, подписки на отчёты, алерты.

Рыночная ниша

WB-Irrigation занимает уникальную нишу на пересечении «промышленное оборудование» + «DIY smart home» + «open source». Коммерческие контроллеры (Rachio, Hydrawise) — closed source с ограниченным hardware. OpenSprinkler — open source, но без промышленного hardware.

WB-Irrigation — единственная open-source система автоматического полива на промышленном DIN-rail контроллере с Modbus/MQTT и observed state verification.

  • ❌ 728 catch-all except — вдвое больше, чем было

Главная рекомендация:

Перестать добавлять новые фичи на 2-3 недели. Провести «Фазу стабилизации»: убрать дублирование, заменить 100+ catch-all except, добавить rate limiting, разбить крупные файлы. Это даст базу для устойчивого роста до 7-8/10.

Для MVP одного разработчика с AI-ассистентом — это достойный результат. Проект решает реальную задачу, работает на боевом оборудовании, и имеет уникальную нишу, которую не закрывает ни один коммерческий аналог.
