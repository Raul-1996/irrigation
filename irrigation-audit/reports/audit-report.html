TL;DR

Система работает: прод держит uptime 13.5 дней, RSS стабильный 224 MB, CPU 0.27%, функциональный UX полива 24 зон закрыт. Это не архитектурная катастрофа и не аварийная ситуация — это незавершённый рефакторинг с одним физически-опасным классом багов (БД↔hardware расхождение) и одним громким процессным провалом (CI/CD никогда не запускался на refactor/v2, логи приложения 13 дней пустые, что делает диагностику любого инцидента слепой).

Три больших пункта до конца квартала:

1Reconciliation БД↔клапан. Сейчас его нет. При power-loss или MQTT-disconnect клапан может остаться открытым без ведома приложения — пролитая вода, сухой ход насоса.
2Логирование. app.log 0 байт 13 дней (bug #4 автора подтверждается) — любая будущая проблема расследуется «по воздуху». Это первое, что нужно починить — 4 часа работы.
3SSE-зомби. Фронт отключил, бэк-хаб жив, браузер бесконечно реконнектится на эндпоинт, возвращающий 204 (~28 800 запросов/сутки на клиента). Решение архитектора — выпилить; подтверждение владельца блокирует 5 мастер-пунктов.

Остальное (CSRF-exempt API на публичном URL, 8 прямых sqlite3.connect в обход репозиториев, scheduler-god-module с 89 LOC дублем, WCAG 2.2 Level A fail, 113 KB dead JS, 55 XPASS тестов в limbo) — чинится поэтапно за 3–4 месяца параллельно с регулярной работой.

Общая оценка зрелости: автор ставит 5/10, архитектор 4.5/10, я соглашаюсь с 4.5/10 с оговоркой: для домашнего контроллера полива эта оценка избыточно строгая — она применима к enterprise-мерке. Для домашнего single-user scope разумнее читать её как «работает, но требует профилактики до осеннего сезона».

3. ⚠️ Физические риски (elevated priority)

Раздел про воду, клапаны и насос — где программная ошибка становится материальным ущербом. Вынесен отдельно: в home-irrigation классе систем это не «Severity-1», а единственная Severity-0 категория.

PHYS-1 — Расхождение БД↔hardware по состоянию зоны CRIT L P0

Cross-ref: CRIT-R1 + CRIT-R3 + CRIT-RC1 + CRIT-CC1 + HIGH-R2 → master MASTER-C1

Что: в приложении нет механизма сверки «клапан в БД считается закрытым — он физически закрыт?». Четыре конкретных сценария пролива:

  1. Power loss посреди полива (CRIT-RC1). Процесс падает, NO-клапаны остаются открытыми. При старте boot_sync полагается на БД, а не на GPIO echo. Если stop-job в MemoryJobStore потерялся — нет таймера, который зону закроет.
  2. MQTT-disconnect в момент stop-команды (CRIT-R1). publish(..., qos=1) без wait_for_publish() + без подтверждения от контроллера = команда «stop» могла не доехать. Приложение отмечает state='off' оптимистично.
  3. Race watchdog vs user-stop (CRIT-CC1). Пользователь нажал Stop, одновременно watchdog cap-time достиг лимита. Два UPDATE на zones без optimistic lock — старое «on» может перезатереть «off».
  4. Fire-and-forget verify (HIGH-R2). services/observed_state.py использует verify_async. Успех/провал verify не влияет на state и не ведёт к alert.

Где: services/zone_control.py (48% coverage), routes/zones_watering_api.py (49%), services/observed_state.py, services/watchdog.py, services/app_init.py, irrigation_scheduler.py:67-155 (второй boot-sync), таблица zones — нет полей desired_state/commanded_state/observed_state/desired_version.

Что делать (направление):

  • Ввести state machine desired → commanded → observed → confirmed (target-state §3).
  • Subscriber на wb-irrigation/zone/+/observed → UPDATE observed_state.
  • APScheduler-job reconcile_tick каждые 10 сек: desired vs observed, retry через command_id, после 3 попыток — state='fault' + алерт.
  • boot_reconcile при старте: retained MQTT snapshot, safety-first policy.
  • Optimistic lock: UPDATE zones SET ... WHERE id=? AND desired_version=?.

PHYS-2 — Scheduler теряет one-shot stop-jobs при рестарте CRIT S P0

Cross-ref: HIGH-R3 → master MASTER-H10

Что: APScheduler настроен на SQLAlchemyJobStore, но SQLAlchemy не установлен на проде. Тихий fallback на MemoryJobStore — все задачи теряются при systemctl restart.

Самый опасный класс — zone_stop:N (one-shot на конец полива). Если рестарт в середине полива — stop-job исчез, клапан открыт.

Что делать: SQLAlchemy в requirements.txt (+~3 MB RSS), отдельный jobs.db, coalesce=True, misfire_grace_time=300. НО не единственная защита — critical one-shot jobs восстанавливаются через boot_reconcile (PHYS-1).

PHYS-3 — Float-sensor монитор обходит репозиторный слой HIGH S P1

Cross-ref: DB-006 частный случай

Что: services/float_monitor.py использует прямой sqlite3.connect() в обход BaseRepository._connect(). PRAGMA foreign_keys=ON, synchronous=NORMAL, busy_timeout=30000 не применяются. Float-sensor — safety-critical сенсор (защита от сухого хода насоса).

Риск: write float-event на busy-lock SQLite упадёт быстро без busy_timeout, reconciler не узнает что бак пустой.

PHYS-4 — Weather-decision идёт мимо репозиториев, 7 прямых connect HIGH M P2

Cross-ref: DB-006 основной, MASTER-L1, MASTER-H3

Что: services/weather.py (1404 LOC god-module) содержит 7 прямых sqlite3.connect(). Weather-adjustment влияет на объём полива (ratio может быть > 1). В комбинации с PHYS-1 + PHYS-2 — пролив с мультипликатором.

Что делать: db/weather.py репозиторий; разбить на domain/weather/decision.py (pure) + infrastructure/weather_http + application/weather_service.

PHYS-5 — Manual API без confirm / rate-limit HIGH S P1

Cross-ref: SEC-002, SEC-003 → master MASTER-H6

Эндпоинты /api/zones/N/start, /stop, /api/programs/N/run — CSRF-exempt, без rate-limit, без confirm. В сочетании с публичным URL poliv-kg.ops-lab.dev (CF Tunnel прямо на Flask) — утечка session cookie / XSS = атакующий массово открывает клапаны.

Из security.md (PoC концептуально, не на проде):
curl -X POST https://poliv-kg.ops-lab.dev/api/zones/5/start
curl -X POST https://poliv-kg.ops-lab.dev/api/groups/1/start-from-first
curl -X POST https://poliv-kg.ops-lab.dev/api/emergency-stop

Что делать: CF Access Application (OAuth) — см. вопрос §9.3. flask-limiter 30 rpm/user на start/stop. Убрать CSRF-exempt с session-authenticated эндпоинтов.

4. Консолидированные master-findings

Находки сгруппированы по корневой причине, не по агенту. Один root-cause = одна мастер-точка. Severity откалибрована под домашний single-user scope. C — чинить эту неделю, H — этот месяц, M — этот квартал, L — когда-нибудь. Effort: S ≤ 1 день, M ≤ 1 спринт, L > 1 спринт.

MASTER-C1: Отсутствует reconciliation-контур БД↔hardware C L P0

Domain: coreDeps: C3, H2, M1Cross-ref: CRIT-R1, CRIT-R3, CRIT-RC1, CRIT-CC1, HIGH-R2, CQ-006, CQ-007, Bug 5.2

Приложение оптимистично пишет в БД «зона открыта/закрыта», не проверяя, что контроллер реально выполнил команду. Нет поля observed_state, нет subscriber'а на echo-топик GPIO, нет периодического reconciler'а, нет optimistic-lock на UPDATE zones.

Где
  • services/zone_control.py (48% coverage, критический путь)
  • routes/zones_watering_api.py (49% coverage)
  • services/observed_state.py (verify_async fire-and-forget — HIGH-R2)
  • services/watchdog.py (race с user-stop — CRIT-CC1)
  • irrigation_scheduler.py:67-155 (второй boot-sync — CQ-006 / BUG 5.2)
  • Таблица zones: нет колонок desired_state, desired_version, commanded_state, observed_state, observed_at, last_command_id
Из sre.md (CRIT-RC1): services/zone_control.py:242 сначала ставит state='stopping', публикует MQTT, ждёт verifier асинхронно, затем без проверки ack ставит state='off' (line 300). Если publish провалился — БД скажет off, hardware останется on, watchdog не сработает (state=off в его выборке).
Что делать
  • State machine desired → commanded → observed → confirmed (target-state §3).
  • Subscriber на wb-irrigation/zone/+/observed.
  • APScheduler job reconcile_tick каждые 10 сек; retry через command_id; после 3 попыток — fault + алерт.
  • boot_reconcile: safety-first, при сомнениях закрыть клапан.
  • Optimistic lock через desired_version++.

MASTER-C2: Диагностический blackout — app.log пустой 13 дней C S P0

Domain: observabilityDeps: none — first thing to fixCross-ref: Bug #4, CRIT-O1, CQ-001..004, CQ-012

app.log 0 байт, прод работает 13.5 дней. Автор пометил как Bug #4 — подтверждаю + расширяю. Две корневые причины:

  1. Handler повешен на named logger 'app', а logger = logging.getLogger(__name__) в модулях использует services.xxx — сообщения идут не в тот logger.
  2. basicConfig(level=WARNING) в irrigation_scheduler.py и scheduler/jobs.py на import-time, до logging_setup.py → перебивает уровень (CQ-012).

Расширение Bug #4: в routes/settings.py, services/locks.py, services/mqtt_pub.py, services/telegram_bot.py нет import logging — logger.info(...) будут NameError'ами, которые не стреляют только потому, что ветки не исполняются. telegram.txt 520 KB без ротации.

Что делать
  • Handler на root logger; setup_logging() вызывать в app.py до импортов scheduler/services.
  • Убрать basicConfig из scheduler/jobs/database.
  • Добавить import logging; logger = logging.getLogger(__name__) в 4 модуля.
  • logrotate для telegram.txt.
  • Structured JSON formatter с correlation_id, zone_id, command_id — во вторую очередь.

MASTER-C3: MQTT — нет owner'а, 18 клиентов, per-request connect, anonymous C M P0

Domain: mqttDeps: none на стартCross-ref: API-01, SP-1, SEC-001
  • services/mqtt_pub.py держит кэш, но routes/system_status_api.py:464-478 создаёт новый mqtt.Client().connect() per-request → ~17 000 TCP-сессий/сутки.
  • 18 отдельных MQTT-клиентов в одном процессе — нет координатора, нет общего reconnect-policy.
  • Все клиенты анонимные (broker без ACL) — любой в LAN может читать/писать любые топики.
  • Нет таксономии, нет идемпотентности команд (QoS=1 + reconnect = double-delivery).
  • publish() без wait_for_publish() — команды могут молча потеряться.
Из security.md (SEC-001, CVSS 9.6): на проде allow_anonymous true на listener'ах 1883 и 18883. Из любой машины в 10.2.0.0/16:
mosquitto_pub -h 10.2.5.244 -p 1883 -t '/devices/wb-mr6c_<ID>/controls/K1/on' -m '1'
Что делать
  • Единственный infrastructure/mqtt/client_pool.py; удалить per-request connect.
  • Свести 18 клиентов к ≤ 3 ролям: pub, sub-monitors, sub-scheduler.
  • MQTT contract v1: namespace wb-irrigation/, zone/{id}/(command|desired|state|observed), command_id (ULID) для идемпотентности через command_log.
  • ACL per identity: irrigation_app, wb_controller, irrigation_tg, telegraf_ro.
  • Dual-publish 2 недели, потом cutover (решение владельца §9.6).

MASTER-C6: WCAG 2.2 Level A — FAIL, 5 критичных блокеров C M P1

BLOCKING 9.1 — SSE: выпил или вернуть?

LoadCredential безопаснее — secrets не видны в systemctl show. EnvironmentFile= проще.

BLOCKING 9.3 — Cloudflare Access: включаем? IdP?

Рекомендация: manual — для embedded с physical hardware безопаснее.
non-blocking 9.5 — TG bot: тот же процесс или отдельный systemd unit?
Рекомендация: оставить один процесс. Вынести позже, если бот станет активнее.
non-blocking 9.6 — MQTT contract: dual-publish 2 недели или cutover?

non-blocking 9.9 — Backup retention: 30 дней на SD достаточно или offsite?

10.4. Definition of Done per master-point

  2. Тесты написаны или обновлены; CI зелёный.
  3. Если затрагивает физику — прогон на staging/mock + smoke-test manual.
  4. Если затрагивает прод-БД — backup до + rollback-план.
  5. Документация: BUGS-REPORT статус, CHANGELOG.

  • Weekly (30 мин): статус master-points, переприоритизация, метрики
  • Monthly: квартальный review — обновить отчёт, DONE в «закрыто»

10.6. Game Day после Волны 2

Симуляция 4 сценариев пролива:

  1. Power loss посреди полива зоны 5
  2. MQTT-disconnect во время stop-команды
  3. Одновременный user-stop + watchdog-stop (race)
  4. verify_async timeout

Цель — reconciler закрывает клапан в каждом сценарии ≤ 30 сек. Если нет — Волна 2 не закрыта.

10.7. Rollback plan

  • Backup БД до начала каждой волны.
  • Tag pre-wave-N на git commit.
  • Если после deploy readyz не зелёный >5 мин — git checkout pre-wave-N && systemctl restart.
  • Для миграций БД (Волна 5, FK): restore БД из backup.
