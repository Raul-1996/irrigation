## Уточнённый scope (от Рауля 2026-06-26)
1. Выход = **коэффициент-множитель** к базовому времени зоны (НЕ мм, НЕ объём, НЕ производительность дождевателей). Только «лить больше/меньше».
2. **Зоны/типы растений НЕ различаем** («ориентируемся на время»). Единый коэффициент на объект. Без per-zone Kc/почвы/корней/TAW.
3. **Датчика дождя НЕТ** — осадки только из Open-Meteo (факт past-day). Заложить опциональный хук под будущий датчик, не реализовывать.
4. Реперная точка нормировки = **климат-норма ET₀** точки (авто, без ввода пользователя).
5. Цель: убрать ночное занижение + внутридневные скачки (корень — мгновенная погода в момент старта зоны).
6. **Флаг-переключатель** H1 ↔ баланс — обязателен.
7. ET₀ не корректировать локальным temp/hum.
8. Anti-drift обязателен (clamp, reset по дождю, ночной пересчёт по факту, ручная корректировка + лог).

## 1. Концепция и формула — вариант A (нормированный скользящий ET-дефицит к климат-норме)

**Почему A, не B (псевдо-Dr в мм):** B тайком протаскивает мм-баланс — для clamp `Dr∈[0,TAW]` нужен TAW в единицах, а перевод полива в пополнение Dr требует производительности спринклера (запрещено scope). B честен только с влагомером/дождемером. A работает в относительных величинах → идеально ложится на «множитель к базовому времени». Базовое время зоны = «правильно при норме»; множитель = «насколько период суше/влажнее нормы». Никаких мм. Honest by construction.

**Поправка:** дефицит копим **скользящей суммой за окно N дней** (не с начала сезона) — встроенное забывание = anti-drift без Калмана (нечем калибровать).

**Формула** (раз в сутки, ночной job, по факту прошедших суток d из Open-Meteo past_days):
```
need_d    = ET0_fact_d × Kc          # Kc = 1.0 (см. ниже)
supply_d  = P_eff_d = max(0, P_fact_d − INTERCEPT_MM)   # INTERCEPT≈4 мм (перехват газоном)
deficit_d = need_d − supply_d

D_win    = clamp( Σ deficit_d (за окно N) , 0 , WINDOW_CAP ) + manual_offset
Norm_win = Σ (ET0_norm_d × Kc) ≈ N × et0_norm_daily

coef = clamp( 100 × D_win / Norm_win , COEF_MIN , COEF_MAX )    # напр. [50,150]
```

**Сокращение Kc:** Kc в числителе (need) и знаменателе (Norm) почти сокращается → формальное подтверждение «зоны не важны». В v1 **Kc=1.0 жёстко**. Снимает весь блок Kc-таблиц из v1. Осадки (supply) не умножаются на Kc — физически верно (дождь гасит потребность независимо от Kc).

**Reset по дождю (saturation):** если `P_eff_d ≥ RAIN_RESET_MM` (≈12 мм) — обнулить накопленный дефицит до дня дождя → coef падает к минимуму. Разрывает «память засухи» после промокания.

**Прогноз осадков в баланс НЕ входит** — только факт past-day (урок Rachio pre-credit bug).

### Поведение в момент полива
Coef берётся из суточного состояния (пересчитан прошлой ночью), **константен весь день** → утренний и вечерний полив получают ОДИН coef. Это лечит оба бага.

**Числовые примеры** (N=3, et0_norm=5, Kc=1, INTERCEPT=4, COEF∈[50,150]):
- Жара 3 дня (ET0=6.5×3, без дождя): deficit=19.5, Norm=15 → **coef=130%**.
- Тёплая влажная ночь (дни умеренные ET0≈4.5): deficit≈13.5, Norm=15 → **coef=90%**. Старый движок ночью дал бы ~50% (ночное занижение). 
- Скачок: два полива в день при разной мгновенной погоде → новый даёт **один coef весь день** (старый: 70%→120% внутридневной скачок).
- Сильный дождь 18 мм (P_eff=14≥12): reset → **coef=50%**.

## Доработки по адверсариальной критике (2026-06-26) — заменяют соответствующие места выше

Критик подтвердил: каркас здоров (отказ от мм-баланса B, отдельный history-запрос, surgical-границы H1/safety-skip — верны и подтверждены кодом). Но 4 блокера + правки. Фиксы:

### БЛОКЕР 1 — изоляция history-запроса от кэша/парсера H1 (cache-collision)
`weather_cache` ключуется ТОЛЬКО по `(round(lat,4),round(lon,4))` (`cache.py:73-78,145-147`), `WeatherService` — синглтон. Если `fetch_history` сохранит ответ через общий `_save_cache`, H1 получит ответ с `past_days` и `daily[0]`≠сегодня → молчаливая поломка (`models.py:107` берёт `daily[...,0]`).
**Фикс (инвариант спеки):** `fetch_history(lat,lon,past_days)` — отдельная функция в `client.py`, **НЕ трогает** `weather_cache`/`_save_cache`/`get_weather()`, **НЕ парсится** через `WeatherData._parse`. Свой минимальный парсер daily-массивов, без кэша (раз в сутки кэш не нужен). H1-путь и балансовый путь не делят ни кэш, ни парсер.

### БЛОКЕР 2 — убрать жёсткий reset, осадки только как кредит (ложный reset → засуха)
QPF FAR до 84% в горах: ложный «дождь 18мм» от OM → жёсткий reset обнулит дефицит → недополив в жару → ущерб газону (необратим). Жёсткий reset был ИЗБЫТОЧЕН: осадки уже вычитаются в `deficit_d = need_d − supply_d`.
**Фикс:** убрать жёсткий reset и параметр `rain_reset_mm`. Осадки входят ТОЛЬКО как кредит через `supply_d` в дневной дефицит, накапливаются в окне. Большой реальный дождь даст большой отрицательный `deficit_d` → сумма окна уйдёт вниз → нижний `clamp(D_win,0,…)` обеспечит «насыщение». Один ложный день не стирает память окна (другие дни остаются) — градуальная, а не бинарная реакция. Проще И безопаснее.

### БЛОКЕР 3 — специфицировать норму (иначе нормировка вырождается)
Если норма = среднее за то же окно N=3, что и дефицит → `D_win/Norm_win ≈ (need−supply)/need` → coef залипает у 100, не реагирует на жару. Норма обязана быть из ДРУГОГО (длинного) горизонта.
**Фикс:** `et0_norm_daily` = **EMA ET₀ за ~30 дней** (α≈2/(30+1)≈0.065), обновляется каждым ночным job: `norm = α·et0_fact + (1−α)·norm`. Bootstrap: первый запуск тянет `past_days≈30`, среднее → стартовая норма. Дефицит — окно 3 дня (быстрый), норма — 30 дней (медленный, сезонный) → отношение = «насколько сейчас жарче нормы», не вырождается. Guard: истории <7 дней / норма не готова → coef=100. На смене сезона EMA-30 запаздывает ~2 нед (приемлемо; v1.1 — Climate API за точной сезонной кривой).

### БЛОКЕР 4 — пустой/неполный буфер → coef=100 (не 50)
Формула `clamp(0,50,150)=50` на старте режет полив вдвое; проза говорит 100. Противоречие.
**Фикс (явное правило):** если `deficit_buffer` короче N полных дней ИЛИ норма не готова → `coef=100` (нейтрально), формулу НЕ применять. Только при полном окне + готовой норме считать `100×D_win/Norm_win`.

### Средние правки (в тело спеки)
- **Stale coef (#5):** в `get_effective_coefficient` — если `last_recalc_date` старше 2 дней (job не отрабатывал: сервис лежал / OM недоступен N ночей) → **fallback на H1 `get_coefficient()`** (не применять протухший баланс) + лог-предупреждение. Новый параметр `stale_fallback_days="2"`.
- **Race settings (#6):** `set_setting_value` неатомарен по нескольким ключам (`db/settings.py:27`). Писать `coef_cached` **последним** (после buffer/date/norm) — его появление гарантирует консистентность остального.
- **TZ (#7):** зафиксировать допущение «локальный TZ контроллера == TZ объекта» (для .244 совпадает). Брать последний **завершённый** день (отступ, НЕ «сегодня» — OM может отдать неполные текущие сутки в зависимости от часа запроса).
- **WINDOW_CAP (#8):** убрать — `coef_max=150` уже ограничивает выход, cap дублирует.
- **Лаг 1 день (#9):** осознанный trade-off — H2 реагирует на резкую смену погоды со след. суток (баланс пересчитан ночью), тогда как H1 реагировал мгновенно (ценой багов скачков). Для инерционной почвы приемлемо. **Проговорить Раулю явно.**

### Упрощения (Karpathy)
- **Убрать `manual_offset` из v1** — лишняя ручка (тихо искажает баланс вечно, +поле/валидация/UI/колонка). Не нужна для починки ночи+скачков. → v1.1.

- **Shadow-режим — сделать обязательным** шагом ввода (не опцией): единственный способ поймать дрейф/ложь нормы до боевого полива.
