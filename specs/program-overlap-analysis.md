> Версия кода: текущая (main branch)

## Секция 1: Текущая защита от пересечений

### 1.1 Проверка конфликтов при создании/редактировании программы (UI/API)

**Файлы:** `routes/programs_api.py`, `db/programs.py` → `check_program_conflicts()`

При создании (`POST /api/programs`) и обновлении (`PUT /api/programs/<id>`) программы вызывается `db.check_program_conflicts()`. Также есть отдельный эндпоинт `POST /api/programs/check-conflicts` для превью конфликтов из UI.

**Алгоритм `check_program_conflicts()`:**

1. Загружает все программы из БД (кроме текущей при редактировании)
2. Для каждой программы проверяет:
   - Есть ли общие дни недели (`common_days`)
   - Есть ли общие зоны (`common_zones`) **ИЛИ** общие группы (`common_groups`)
   - Есть ли пересечение по времени (интервалы `[start, start+total_duration]`)
3. Длительность программы = сумма `duration` всех зон из БД

**Проблемы проверки:**
- **Используются BASE durations из БД**, а не weather-adjusted. Если коэффициент погоды >100%, реальная длительность будет больше, и конфликт может возникнуть в рантайме даже если проверка показала "нет конфликтов"
- **Проверка блокирующая только в UI** — возвращает `has_conflicts: true`, но НЕ запрещает сохранение. Фронтенд должен сам решить, показывать ли предупреждение. При вызове через API без фронтенда (или при игнорировании конфликта) программа будет сохранена
- Нет учёта `early_off_seconds` (раннее выключение 0-15 сек) — минимальная погрешность
- Нет учёта station delay между зонами

### 1.2 Runtime-защита (APScheduler)

**Файл:** `irrigation_scheduler.py`

#### `max_instances=1` на CronTrigger job

```python
_kwargs = dict(
    ...
    max_instances=1,
)
job = self.scheduler.add_job(job_run_program, trigger, **_kwargs)
```

Каждая программа регистрируется как отдельный CronTrigger job с `id=f"program:{program_id}:d{day}"`. Параметр `max_instances=1` означает, что **одна и та же программа** не запустится дважды одновременно (APScheduler пропустит второй запуск). 

**НО:** Это защищает ТОЛЬКО от повторного запуска _той же самой_ программы. Две _разные_ программы (`program:1:d1` и `program:2:d1`) запустятся параллельно без каких-либо проверок.

#### `misfire_grace_time=3600`

Если APScheduler пропустил запуск (например, сервис был down), программа запустится в течение 1 часа после планового времени. Это может привести к неожиданному наложению с другой программой.

#### Нет глобальной очереди программ

В отличие от профессиональных контроллеров (Hunter, Rain Bird — см. Секцию 5), **нет механизма stacking/queueing**. Если программа A стартует в 06:00 и длится до 06:45, а программа B стартует в 06:30 — обе запустятся параллельно в отдельных потоках.

### 1.3 `group_cancel_events`

**Тип:** `Dict[int, threading.Event]` — по одному Event на group_id.

**Назначение:** Позволяет прервать текущий последовательный полив группы.

**Как работает:**
1. При `start_group_sequence()` создаётся `cancel_event = threading.Event()`
2. В цикле `_run_group_sequence()` и `_run_program_threaded()` каждую секунду проверяется `cancel_event.is_set()`
3. При `cancel_group_jobs()` вызывается `event.set()`, что приводит к:
   - Немедленному выходу из цикла ожидания текущей зоны
   - Пропуску оставшихся зон в последовательности
   - Вызову `stop_all_in_group()` для принудительной остановки всех зон

**Ограничения:**
- Один Event на группу, не на программу. Если две программы работают с одной группой — отмена затронет обе
- Event очищается (`clear()`) и удаляется в `finally` блоке `_run_group_sequence`, но НЕ в `_run_program_threaded`

### 1.4 Peer-зоны OFF перед стартом

В `_run_program_threaded()`, перед запуском каждой зоны:

```python
# БЕЗУСЛОВНО выключаем все зоны этой группы перед стартом текущей
group_peers = [z for z in all_zones if z['group_id'] == group_id and int(z['id']) != int(zone_id)]
for gz in group_peers:
    publish_mqtt_value(server, t, '0', ...)
    self.db.update_zone(int(gz['id']), {'state': 'off', 'watering_start_time': None})
```

**Что это делает:** Перед включением каждой зоны выключает все другие зоны в той же группе. Это гарантирует, что в одной группе (= на одном насосе) работает максимум одна зона.

**Где работает:** Только в `_run_program_threaded()`. В `_run_group_sequence()` это не нужно, т.к. последовательность и так гарантирует одну зону за раз.

**Проблема:** Если две программы запускают зоны из одной группы параллельно, каждая будет выключать зону другой. Результат — хаотичное переключение: программа A включает зону 1, программа B выключает зону 1 и включает зону 3, программа A выключает зону 3 и включает зону 2, и т.д.

### 1.5 `exclusive_start_zone()` (services/zone_control.py)

**Централизованный старт зоны** с сериализацией через `group_lock()`:

```python
with group_lock(group_id):
    # Start current zone
    # Stop all peers in group (parallel via ThreadPoolExecutor)
```

Использует `threading.RLock` per group_id. Гарантирует, что в пределах одного процесса операции старт/стоп для зон одной группы сериализованы. **НО:** Это защищает только от race condition на уровне MQTT-публикаций, а не от логического пересечения программ.

### 1.6 Watchdog и Cap-стопы

- `schedule_zone_hard_stop()` — жёсткий стоп зоны по `DateTrigger` на `planned_end_time`
- `schedule_zone_cap()` — абсолютный лимит работы зоны (по умолчанию 240 мин)
- `schedule_master_valve_cap()` — лимит открытого мастер-клапана (по умолчанию 24 часа)

Эти механизмы — страховка от "залипших" зон, но не защита от пересечений.

### 1.7 `is_program_run_cancelled_for_group()`

В `_run_program_threaded()` перед каждой зоной проверяется, не отменён ли запуск программы для конкретной группы на сегодня:

```python
if _db.is_program_run_cancelled_for_group(int(program_id), today, int(group_id)):
    logger.info(f"Программа {program_id}: отменена для группы {group_id} на {today}, зона {zone_id} пропущена")
    continue
```

Таблица `program_cancellations` — ручная отмена конкретного запуска. Не связана с автоматической защитой от пересечений.

## Секция 2: Проблемы

### 2.1 Сценарий: Две программы с пересекающимися временами

**Условия:**
- Программа A: 06:00, зоны [1, 2, 3] (группа 1), суммарно 45 мин
- Программа B: 06:30, зоны [4, 5, 6] (группа 1), суммарно 30 мин

**Что происходит:**
1. 06:00 — APScheduler запускает `job_run_program(A)` в потоке Thread-1
2. Thread-1: включает зону 1, спит 15 мин
3. 06:30 — APScheduler запускает `job_run_program(B)` в потоке Thread-2
4. Thread-2: перед включением зоны 4 выключает все peer-зоны группы 1 → **выключает зону 1!**
5. Thread-2: включает зону 4, спит 10 мин
6. 06:35 — Thread-1 просыпается (зона 1 "отработала"), выключает зону 1 (уже off), включает зону 2 → **выключает зону 4!**

**Результат:** Хаотичное чередование, ни одна зона не получает полного полива. Насос включается/выключается с высокой частотой (потенциально опасно).

### 2.2 Сценарий: Программы с зонами из разных групп

**Условия:**
- Программа A: 06:00, зоны [1(группа 1), 4(группа 2)], суммарно 30 мин
- Программа B: 06:15, зоны [2(группа 1), 5(группа 2)], суммарно 30 мин

**Что происходит:**
- Программы обрабатывают зоны последовательно в одном потоке
- Программа A запустит зону 1 (группа 1), потом зону 4 (группа 2)
- Программа B запустит зону 2 (группа 1), потом зону 5 (группа 2)
- Peer-OFF работает только в рамках одной группы, поэтому зоны из разных групп могут работать одновременно (это нормально)
- НО: зоны из одной группы (1 и 2) попадут в "борьбу" (см. сценарий 2.1)

### 2.3 Сценарий: Погодокоррекция увеличивает длительность

**Условия:**
- Программа A: 06:00, зоны [1, 2, 3], base duration по 15 мин = 45 мин, заканчивается в 06:45
- Программа B: 07:00, зоны [4, 5, 6], та же группа
- Погодный коэффициент: 150% (жара)

**Что происходит:**
1. `check_program_conflicts()` при создании считает: A заканчивается в 06:45, B начинается в 07:00 → **конфликта нет**
2. В рантайме: `_get_weather_adjusted_duration()` возвращает 22.5 → 23 мин на зону
3. Реальная длительность A: 23 × 3 = 69 мин, заканчивается в **07:09**
4. Программа B стартует в 07:00 — **пересечение 9 минут!**

**Результат:** Тот же хаос, что в сценарии 2.1, при том что при создании программ конфликтов не было.

### 2.4 Сценарий: misfire_grace_time сдвигает старт

**Условия:**
- Сервис был перезапущен, программа пропустила плановый старт на 30 минут
- `misfire_grace_time=3600` позволяет запуск до часа после плана

**Что происходит:**
- Программа стартует с опозданием на 30 мин, пересекаясь со следующей программой

### 2.5 Сценарий: recover_missed_runs() создаёт наложение

**Условия:**
- Сервис перезапустился в 06:25
- Программа A должна была стартовать в 06:00 (зоны по 15 мин)
- Программа B стартует в 06:30

**Что происходит:**
1. `recover_missed_runs()` определяет, что программа A ещё не завершена
2. Запускает оставшиеся зоны программы A немедленно
3. В 06:30 APScheduler запускает программу B
4. Обе работают параллельно

### 2.6 Что конкретно ломается

| Проблема | Последствия |
|---|---|
| Две программы на одной группе | Зоны "бьются" за включение, ни одна не получает полный полив |
| Частое переключение клапанов | Износ соленоидов, гидроудары в системе |
| Насос вкл/выкл | Перегрев, сокращение ресурса, потенциальный сухой ход |
| Master valve | Может быть закрыт раньше времени из-за `_delayed_close` другого потока |
| Статистика воды | `zone_runs` перемешиваются, `total_liters` неточен |
| Planned_end_time | Перезаписывается конкурирующим потоком |

## Секция 3: Группы и насосы

### 3.1 Структура

**Группа** = логический набор зон, обслуживаемых одним насосом/мастер-клапаном.

Таблица `groups`:
- `id`, `name`
- `use_master_valve` (0/1) — есть ли мастер-клапан
- `master_mqtt_topic` — MQTT-топик мастер-клапана
- `master_mode` — NC (нормально закрытый, по умолчанию) или NO
- `master_mqtt_server_id` — какой MQTT-сервер
- `use_water_meter` (0/1) — есть ли счётчик воды
- `water_mqtt_topic`, `water_pulse_size`, `water_base_value_m3`

### 3.2 Может ли быть несколько групп?

**Да.** Система поддерживает неограниченное количество групп. В коде нет ограничений. Типичная конфигурация: несколько групп по зонам водоснабжения (разные насосы, разные участки).

Группа `999` — специальная ("Без группы"), не может быть удалена. Зоны в группе 999 не получают master valve логику.

### 3.3 Может ли одна зона быть в нескольких группах?

**Нет.** Зона имеет единственное поле `group_id` (FK → groups.id). Одна зона принадлежит ровно одной группе.

### 3.4 Может ли программа содержать зоны из разных групп?

**Да.** Поле `zones` в таблице `programs` — это JSON-массив zone_id без ограничений по группам. Это ключевой архитектурный момент:
- Программа может включать зоны [1, 2, 3] (группа 1) и [4, 5] (группа 2)
- `_run_program_threaded()` обрабатывает их **последовательно** (не параллельно по группам)
- Peer-OFF выключает только зоны той же группы, а не все зоны программы

### 3.5 Master Valve логика

**Открытие:** В `exclusive_start_zone()` перед публикацией zone ON:
1. Определяется группа зоны
2. Если `use_master_valve=1` — публикуется OPEN на `master_mqtt_topic`
3. Значение зависит от mode: NC → '1' (подать питание = открыть), NO → '0'

**Закрытие:** В `stop_zone()` после публикации zone OFF:
1. Запускается отложенный тред `_delayed_close` (через `MASTER_VALVE_CLOSE_DELAY_SEC = 60` секунд)
2. Через 60 сек проверяет: есть ли хоть одна ON зона в **любой группе** с тем же `master_mqtt_topic`
3. Если нет ON зон → закрывает мастер-клапан
4. Если есть — не закрывает (другая зона ещё работает)

**Потенциальная проблема при пересечении:**
- Поток A завершает зону → запускает _delayed_close
- Через 60 сек проверяет — зон ON нет (поток B уже переключил свою)
- Закрывает мастер-клапан
- Но поток B всё ещё считает, что его зона работает
- Следующая зона потока B включится без мастер-клапана? Нет — `exclusive_start_zone` откроет его заново, но будет кратковременный пропуск давления

### 4.2 Где происходит weather adjustment

| Этап | Метод | Что делает |
|---|---|---|
| Перед программой | `_check_weather_skip()` | Skip целиком (rain/freeze/wind) |
| Перед каждой зоной | `_get_weather_adjusted_duration()` | Корректирует длительность (0-200%) |
| Коэффициент | `WeatherAdjustment.get_coefficient()` | Zimmerman + ET₀ hybrid |

Weather adjustment вызывается **в момент запуска зоны**, а не при планировании. Это значит:
- Коэффициент может отличаться для разных зон одной программы (погода меняется)
- Общая длительность программы непредсказуема на этапе планирования

### 4.3 Где проверяется busy/occupied

**Нигде.** В текущей реализации **нет проверки**, занята ли зона или группа перед запуском. Конкретно:

- `_run_program_threaded()` — НЕ проверяет `zone.state == 'on'` перед стартом
- `_run_group_sequence()` — НЕ проверяет, идёт ли уже полив
- `schedule_program()` — НЕ проверяет, не запущена ли уже программа
- `exclusive_start_zone()` — проверяет `cur_state in ('on', 'starting')` и пропускает повторный старт, но НЕ блокирует запуск новой зоны, если другая зона группы уже ON (она просто выключит старую)

Единственная "проверка" — это peer-OFF, который силой выключает конкурирующие зоны.

## Секция 5: Как это делают профессиональные контроллеры (Hunter, Rain Bird)

### 5.1 Hunter Pro-HC / X2 / Hydrawise

**Program Stacking (очередь):**
- Контроллер **не поддерживает** одновременную работу нескольких станций/зон
- Если программа A и программа B запланированы на одно время — B **ждёт в очереди**, пока A полностью не завершится
- Станции всегда запускаются последовательно, одна за другой

**Station Delay (Inter Zone Delay):**
- Настраиваемая пауза между завершением одной зоны и стартом следующей (1 сек — 8 часов)
- Назначение: 
  - Время на закрытие медленных клапанов (предотвращение гидроудара)
  - Восстановление уровня воды в скважине
  - Управление гидравлической нагрузкой на насос
- Master Valve/Pump остаётся активен во время delay (настраивается)

**Cycle and Soak:**
- Разбивка цикла полива на короткие сегменты с паузами
- Предотвращает сток воды на тяжёлых почвах

**Hydrawise Smart Watering:**
- Автоматическая корректировка длительности на основе погодных данных
- Predictive watering — учитывает прогноз осадков
- Seasonal adjustment — плавное изменение длительности в зависимости от времени года

### 5.2 Rain Bird ESP-ME

**Program Stacking:**
- 4 независимые программы (A, B, C, D), по 6 стартов в день каждая
- Только одна программа работает в любой момент
- При совпадении стартов — вторая программа **ждёт** завершения первой
- Станции внутри программы — строго последовательно

### 5.3 Ключевые отличия от wb-irrigation

| Функция | Hunter/Rain Bird | wb-irrigation |
|---|---|---|
| Program stacking (очередь) | ✅ Встроена | ❌ Нет — параллельный запуск |
| Station delay | ✅ Настраивается | ⚠️ `early_off_seconds` (0-15 сек, только ранний стоп) |
| Overlap detection runtime | ✅ Невозможен по дизайну | ❌ Нет проверки |
| Max simultaneous stations | 1 (жёстко) | Не ограничено |
| Weather-adjusted scheduling | ✅ Учитывает при планировании | ❌ Только в рантайме |
| Cycle and Soak | ✅ Есть | ❌ Нет |
| Master valve smart management | ✅ Интегрировано | ⚠️ Delayed close, но без учёта пересечений |

### 5.4 Рекомендуемый подход: Global Program Queue

Профессиональные контроллеры решают проблему пересечений **архитектурно**: единая глобальная очередь, через которую проходят все запуски. Варианты реализации для wb-irrigation:

1. **Простой вариант: Global Lock per Group**
   - Перед запуском программы захватить lock группы
   - Если lock занят — ждать (с таймаутом)
   - Гарантирует, что одновременно на одной группе работает только одна программа

2. **Полноценный вариант: Program Queue**
   - Глобальная очередь `(group_id, program_id, zones, scheduled_time)`
   - Воркер извлекает из очереди и выполняет последовательно
   - Если время программы B настало, а A ещё не завершена — B ждёт в очереди
   - Опционально: максимальное время ожидания, после которого программа отменяется

3. **Учёт weather-adjusted duration при планировании**
   - При проверке конфликтов (`check_program_conflicts`) учитывать текущий/прогнозный коэффициент погоды
   - Или добавить safety margin (например, +30% к базовой длительности)
