## 1. КРИТИЧЕСКИЕ (must fix перед реализацией)

### К1. Race condition: cancel_group() во время _run_entry() — зоны остаются ON

**Секция спеки:** 2.4, 2.8

**Проблема:** Worker-поток в `_run_entry()` проверяет `cancel_event` в цикле `while remaining > 0`. Но между моментом, когда worker вышел из цикла (cancel detected) и моментом, когда worker вызовет `stop_zone()`, проходит ненулевое время. При этом `cancel_group()` (секция 2.8) вызывает `group_cancel_events[gid].set()`, но в спеке **не описано**, кто именно делает физический OFF зонам при cancel.

**Сценарий:**
1. Worker выполняет зону 2, remaining=300 сек
2. Пользователь нажимает «Стоп группа» → `cancel_group()` → `event.set()`
3. Worker ловит event, выходит из while
4. Worker переходит к `stop_zone()` — но в текущем `cancel_group_jobs()` уже вызван `stop_all_in_group()` параллельно
5. Двойной stop_zone() — идемпотентно, но `_delayed_close` мастер-клапана запускается дважды с разными таймингами

**Решение:** Спека должна явно определить: при cancel_group() **только** worker-поток отвечает за остановку зон (через break из цикла → stop_zone в finally-блоке). `cancel_group()` устанавливает event и **ждёт** завершения worker, а не дублирует stop_all_in_group. Либо наоборот — cancel_group() делает stop_all_in_group(), а worker просто выходит без повторного stop. Но не оба одновременно.

### К2. Deadlock: _global_lock + GroupQueue.lock — не определён порядок захвата

**Секция спеки:** 2.2, 2.7

**Проблема:** Спека описывает два уровня блокировок: `_global_lock` (для создания/удаления GroupQueue) и `GroupQueue.lock` (для доступа к deque). Но **не определён lock ordering**. 

**Сценарий:**
1. Поток A: `enqueue()` захватывает `_global_lock` (чтобы получить/создать GroupQueue), затем `GroupQueue.lock`
2. Поток B: `_worker()` держит `GroupQueue.lock` (в цикле), вызывает `get_all_queues_state()` → пытается захватить `_global_lock` для итерации по всем группам
3. **Deadlock**: A ждёт group.lock (B держит), B ждёт _global_lock (A держит)

**Решение:** Явно зафиксировать в спеке: **всегда сначала _global_lock, потом GroupQueue.lock**. Worker НЕ должен захватывать _global_lock. `get_all_queues_state()` должен копировать dict `_queues` под `_global_lock`, затем итерировать копию, захватывая каждый `GroupQueue.lock` отдельно.

### К3. Насос работает без клапанов при float resume

**Секция спеки:** 3.5

**Проблема:** При возобновлении после float pause спека говорит: «Публикуем MQTT ON (через `exclusive_start_zone`)». Но `exclusive_start_zone()` сначала открывает мастер-клапан (насос), потом включает зону (клапан). Между этими двумя MQTT-публикациями проходит ненулевое время. При float resume **несколько зон** могут возобновляться последовательно — и между ними мастер-клапан уже открыт.

**Но главное:** Спека описывает resume через `exclusive_start_zone()`, который **выключает peer-зоны**. Если FloatMonitor возобновляет зоны 2, 3, 4 одновременно — каждый вызов `exclusive_start_zone()` выключит предыдущие! Зоны будут «бороться» друг с другом — ровно та проблема, которую очередь должна решить.

**Сценарий:**
1. Зоны 2, 3, 4 группы 1 на паузе (float), remaining: 5, 8, 10 мин
2. Уровень восстановлен → FloatMonitor вызывает resume
3. `exclusive_start_zone(2)` → MV ON, zone 2 ON, zone 3 OFF, zone 4 OFF
4. `exclusive_start_zone(3)` → zone 3 ON, zone 2 OFF, zone 4 OFF
5. Зона 2 не доработала remaining!

**Решение:** При float resume **НЕ вызывать exclusive_start_zone** для множественного восстановления. FloatMonitor должен сигнализировать worker-потоку (через event), а worker возобновляет текущую зону и продолжает последовательность. В спеке написано, что worker спит на `float_resume_event.wait()` (секция 3.6), но resume-логика (секция 3.5 пункт 4) описывает прямое включение всех paused-зон — это противоречие. **Worker должен быть единственным, кто включает зоны.**

### К4. Потеря remaining_seconds при перезагрузке контроллера во время float pause

**Секция спеки:** 3.5, 3.7

**Проблема:** `float_pause_state` — это in-memory dict (`{zone_id: remaining_seconds}`). Если контроллер перезагрузится во время float pause:
- remaining_seconds теряется
- Зоны в БД имеют `state='paused'`, но программа потеряна из очереди (in-memory deque)
- `stop_on_boot_active_zones()` выключит зоны с state='paused'? Не определено — текущий код проверяет `state in ('starting', 'on', 'stopping')`, `paused` не входит

**Сценарий:**
1. Зона 3 на паузе, remaining=12 мин
2. Контроллер перезагрузился (watchdog OOM, отключение питания)
3. Сервис стартует → `stop_on_boot_active_zones()` НЕ выключает paused-зоны
4. Зоны висят в state='paused' навечно. Или наоборот — если добавить 'paused' в boot cleanup — remaining теряется

**Решение:** 
1. Сохранять `remaining_seconds` в БД (поле `pause_remaining_seconds` в zones) при паузе
2. При boot: зоны с state='paused' → безусловно OFF (нет гарантии уровня воды после перезагрузки)
3. `recover_missed_runs()` должен учитывать paused-зоны при восстановлении
4. Добавить в спеку секцию «Восстановление после перезагрузки при float pause»

### К5. weather_adjustment применяется при enqueue, а не при выполнении

**Секция спеки:** 2.4, 5.3

**Проблема:** Спека в диаграмме (секция 5.4) показывает «Weather adjust duration» внутри `_run_entry()` перед каждой зоной — это правильно. Но если программа ждала в очереди 90 минут (max_wait=120), погодный коэффициент за это время мог измениться с 150% до 80% (облака пришли, температура упала). Вопрос: спека не уточняет, применяется ли коэффициент **в момент enqueue** или **в момент фактического старта зоны**.

**Сценарий:**
1. 06:00 — Prog A enqueue, коэфф=150%
2. 06:30 — Prog B enqueue (та же группа), коэфф=150%
3. Prog A работает до 07:07 (45 мин × 1.5 ≈ 68 мин)
4. 07:07 — Prog B стартует, но к этому времени облака, коэфф=80%
5. Prog B должна использовать 80%, а не 150%

**Решение:** Спека должна **явно** зафиксировать: weather coefficient вычисляется **в момент фактического старта каждой зоны** (внутри `_run_entry`), а не при enqueue. Это уже подразумевается в диаграмме (секция 5.4), но нужно отдельное явное утверждение.

### К6. max_wait_time vs float pause — программа может быть необоснованно отменена

**Секция спеки:** 2.6, 3.5

**Проблема:** Таймер max_wait_time отсчитывается от `enqueued_at`. Но если текущая программа на паузе по поплавку, все программы в очереди продолжают «тикать» max_wait_time. Если float pause длится 25 минут, а программа B уже ждала 100 минут в очереди, то при max_wait=120 мин — B получит EXPIRED через 20 минут, хотя пауза — объективная причина.

**Сценарий:**
1. 06:00 — Prog A стартует (длительность 90 мин)
2. 06:30 — Prog B enqueue (max_wait=120 мин → дедлайн 08:30)
3. 07:00 — Float pause (30 мин)
4. 07:30 — Float resume, Prog A продолжает (осталось 30 мин)
5. 08:00 — Prog A завершается. Prog B начинает. 
6. 08:30 — Prog B ждала ровно 120 мин → EXPIRED? Или нет, т.к. она уже RUNNING?

**Проблема 2:** Спека проверяет max_wait_time при **извлечении** из очереди (секция 2.4: «Берёт WAITING entry → Проверяет max_wait_time → если истёк → EXPIRED, skip»). Но что если entry уже извлечена (state=RUNNING), а float pause начинается? `remaining` в worker уменьшается только при sleep(1), float pause **замораживает** worker. max_wait_time уже не проверяется — entry RUNNING.

**Решение:** max_wait_time должен **приостанавливаться** на время float pause. Добавить поле `wait_paused_seconds` в QueueEntry. Либо: max_wait_time считать только время фактического ожидания в WAITING (не RUNNING/PAUSED).

## 2. СЕРЬЁЗНЫЕ (should fix, риск проблем в проде)

### С1. Программа с зонами из 3+ групп — несогласованное завершение

**Секция спеки:** 2.5

**Проблема:** Программа «Утро» с зонами из групп 1, 2, 3 создаёт 3 QueueEntry. Каждый выполняется независимо. Но нет механизма отслеживания: «все 3 сегмента программы X завершились». Лог `program_finish` — кто его создаёт? Раньше его создавал `_run_program_threaded()` в конце. Теперь `job_run_program()` возвращается немедленно после enqueue (секция 2.4). Лог «программа завершена» никто не создаст.

**Сценарий:**
1. Программа «Утро» → 3 entry в 3 группы
2. Группа 1 завершилась за 20 мин
3. Группа 2 ждала 40 мин в очереди, потом 15 мин работала
4. Группа 3 завершилась за 10 мин
5. «Программа Утро завершена» — когда? Никто не отслеживает

**Решение:** Добавить `ProgramCompletionTracker`:
- При enqueue: регистрировать `{program_id: set(entry_ids)}`
- При завершении entry: убирать из set
- Когда set пуст → лог `program_finish` + Telegram

### С2. Один entry_id для программы, запущенной дважды в разные дни

**Секция спеки:** 2.2

**Проблема:** `entry_id = f"{program_id}:{group_id}:{timestamp}"` — OK, если timestamp достаточно гранулярный. Но если используется `int(time.time())`, то два enqueue в одну секунду (recover + scheduled) дадут одинаковый entry_id.

**Решение:** Использовать `uuid4()` или `f"{program_id}:{group_id}:{time.monotonic_ns()}"`.

---

### С3. MQTT-брокер недоступен при float resume — зоны не включатся

**Секция спеки:** 3.5

**Проблема:** При float resume FloatMonitor публикует MQTT ON для зон. Если MQTT-брокер отвалился (перезагрузка, сеть) — publish_mqtt_value может бросить исключение или молча потерять сообщение (QoS 2 с paho-mqtt не гарантирует доставку при переподключении).

**Сценарий:**
1. Float pause → зоны OFF
2. MQTT-брокер перезагружается
3. Float восстановлен → FloatMonitor пытается опубликовать ON
4. paho-mqtt не переподключился → publish fails
5. Зоны остаются OFF, worker думает что они ON, remaining тикает

**Решение:** 
1. `_run_entry()` worker должен **верифицировать** что зона реально включилась (уже есть `state_verifier.verify_async`)
2. FloatMonitor при resume должен делать retry с экспоненциальным backoff
3. Если MQTT down > N секунд при resume — аварийный стоп (нельзя поливать, если не контролируем клапаны)

### С4. SQLite write contention при параллельных логах и zone_run updates

**Секция спеки:** 7.3, 7.4

**Проблема:** При 3 группах работающих параллельно, каждая пишет в:
- `program_queue_log` (при каждом изменении state)
- `float_events` (при паузе/resume)
- `logs` таблица (add_log)
- `zones` таблица (update_zone)
- `zone_runs` таблица (create/finish zone_run)

SQLite WAL позволяет concurrent reads, но writes всё ещё сериализованы. При 3 worker-потоках + FloatMonitor + MQTT callbacks — потенциально 5-6 потоков пишут одновременно.

**Сценарий:**
1. 3 группы работают параллельно
2. Float pause на группе 1 → запись float_event + update 8 zones
3. Группа 2: zone_run finish → write zone_runs + update zone
4. Группа 3: zone start → write logs + create zone_run + update zone
5. SQLite: `database is locked` (default timeout 5 sec)

**Решение:**
1. Увеличить `sqlite3.connect(timeout=30)` (текущий дефолт 5 сек)
2. Batch writes: FloatMonitor должен делать один `executemany()` для обновления всех зон, а не N отдельных UPDATE
3. Использовать WAL2 mode если доступен, или `PRAGMA busy_timeout=30000`

### С5. wb-rules watchdog конфликтует с FloatMonitor — «пинг-понг» реле

**Секция спеки:** 3.9, 3.10

**Проблема:** Спека описывает решение через `group_{N}_tripped` топик. Но есть временное окно:
1. Поплавок OFF
2. wb-rules **немедленно** выключает реле (миллисекунды)
3. FloatMonitor получает MQTT float=OFF через ~100ms
4. FloatMonitor начинает паузу, пытается выключить зоны → они уже OFF
5. ОК до этого момента. Но:
6. wb-rules watchdog публикует `tripped=1`
7. FloatMonitor получает `tripped=1` — уже в паузе, ОК
8. **Поплавок ON** (уровень восстановился)
9. FloatMonitor хочет resume → публикует MQTT ON для зон
10. wb-rules watchdog видит поплавок ON → **сбрасывает tripped?** Спека не описывает!
11. Если wb-rules НЕ сбрасывает tripped — FloatMonitor не сможет resume (видит tripped=1)
12. Если wb-rules сбрасывает tripped — нужен порядок: сначала reset tripped, потом FloatMonitor resume

**Решение:** Спека должна описать полный lifecycle `tripped`:
- wb-rules: `tripped=1` при float OFF, `tripped=0` при float ON
- FloatMonitor: при tripped=1 → force pause. При tripped=0 — НЕ автоматический resume (только по собственной логике float ON + debounce)
- wb-rules НИКОГДА не включает зоны обратно (только OFF)

### С6. Дребезг поплавка на границе debounce — false resume

**Секция спеки:** 3.4, 3.5

**Проблема:** Debounce 5 сек. Что если поплавок дребезжит с периодом ровно 5 сек? OFF(5s) → ON(5s) → OFF(5s)...

**Сценарий:**
1. Поплавок OFF → debounce timer 5 сек
2. t=5s: подтверждение → пауза, зоны OFF
3. t=6s: поплавок ON → debounce timer 5 сек
4. t=11s: подтверждение → resume, зоны ON, MV ON
5. t=12s: поплавок OFF → debounce timer 5 сек
6. t=17s: подтверждение → пауза, зоны OFF, MV OFF
7. Цикл каждые 11 сек: ON/OFF/ON/OFF — гидроудары, износ соленоидов, стресс насоса

**Решение:** Добавить **hysteresis** или **min_on_time** после resume:
- После resume: минимум 60 сек работы перед повторной паузой
- Или: после pause — минимум 30 сек перед resume (cooldown)
- Или: после 3 срабатываний за 5 минут → аварийный стоп + Telegram «поплавок нестабилен»

### С7. Две группы с одним насосом (ошибка конфигурации) — нет валидации

**Секция спеки:** 2.1

**Проблема:** Спека предполагает «группа = насос». Но пользователь может создать 2 группы с одним и тем же master_mqtt_topic (т.е. одним насосом). Очередь per-group не защитит — обе группы будут работать параллельно, но физически на одном насосе.

**Сценарий:**
1. Группа 1: master_topic = `/devices/wb-mr6c/controls/K1`, зоны 1-6
2. Группа 2: master_topic = `/devices/wb-mr6c/controls/K1` (ОШИБКА — тот же насос!), зоны 7-12
3. Обе группы стартуют одновременно → один насос обслуживает 2 зоны параллельно → давление падает

**Решение:** Валидация в UI/API: при сохранении группы проверять уникальность `master_mqtt_topic` среди всех групп. Если дубль — WARNING: «Этот мастер-клапан уже используется группой X».

### С8. shutdown() во время float pause — возможна утечка потока

**Секция спеки:** 2.7, 3.6

**Проблема:** При shutdown worker спит на `float_resume_event.wait()`. Спека описывает `stop_event` per-group для graceful shutdown. Но в коде ожидания (секция 3.6) worker проверяет shutdown ПОСЛЕ возвращения из `float_monitor.wait_for_resume_or_timeout()`. Если этот метод блокирует бесконечно — worker не завершится.

**Сценарий:**
1. Зоны на float pause
2. systemd посылает SIGTERM → scheduler.stop() → _shutdown_event.set()
3. Worker в `float_monitor.wait_for_resume_or_timeout()` — заблокирован
4. systemd ждёт TimeoutStopSec (обычно 90 сек), потом SIGKILL
5. Зоны остаются в state='paused' в БД

**Решение:** `wait_for_resume_or_timeout()` должен принимать `shutdown_event` и делать `Event.wait()` с проверкой нескольких событий (через `threading.Event` composition или periodic check). Либо: использовать `threading.Condition` с notify при любом событии (resume, timeout, shutdown, cancel).

### С9. deque maxlen не указан — потенциальный memory leak

**Секция спеки:** 2.2

**Проблема:** `deque()` создаётся без maxlen. Если по какой-то причине worker не стартует или застрял — enqueue будет добавлять entries бесконечно.

**Сценарий:**
1. Worker застрял на MQTT publish (timeout 30 сек)
2. APScheduler продолжает fire программы каждые 15 мин (разные дни/программы)
3. 10 программ × 2 группы = 20 entries за час
4. При max_wait=120 мин — до 40 entries в deque
5. Не критично по памяти, но entries копятся и все получат EXPIRED

**Решение:** `deque(maxlen=50)` в спеке упомянут в задании, но в самой спеке maxlen **не указан**. Добавить: `deque(maxlen=MAX_QUEUE_SIZE)`, при переполнении — reject с логом и Telegram. MAX_QUEUE_SIZE = 20 (достаточно для любого разумного сценария).

### С10. _run_entry() — нет обработки исключений при старте зоны

**Секция спеки:** 5.3, 5.4

**Проблема:** В текущем `_run_program_threaded()` есть `try/except` вокруг `exclusive_start_zone()`. Спека не описывает обработку ошибок в `_run_entry()`. Если `exclusive_start_zone()` бросит исключение (MQTT down, SQLite locked) — worker-поток может умереть, а entry останется в state=RUNNING навсегда.

**Решение:** `_run_entry()` должен быть обёрнут в `try/finally`:
- `finally`: entry.state = COMPLETED/FAILED, зоны OFF, лог
- Unhandled exception → entry.state = 'failed' (новый state), лог, Telegram, worker продолжает со следующей entry

## 3. ЗАМЕЧАНИЯ (nice to have, улучшения)

### З1. MQTT publish storm при float pause/resume 24 зон

**Секция спеки:** 3.5

При паузе группы с 24 зонами — 24 MQTT OFF + 1 MV OFF = 25 публикаций. При resume — 24 ON + 1 MV ON = 25 публикаций. На ARM контроллере с paho-mqtt это ~2-3 секунды.

**Решение:** Пауза и resume зон последовательно с 50ms задержкой (уже есть в `stop_all_in_group`). Но при resume — **worker и так включает зоны последовательно** (одна за другой), так что storm на resume маловероятен. На паузе — добавить `time.sleep(0.05)` между OFF.

### З2. program_queue_log растёт неограниченно

**Секция спеки:** 7.3

Нет политики ротации/очистки. При 10 программах × 7 дней × 2 группы = 140 записей/неделю. Через год — 7000+ записей. Не критично, но...

**Решение:** Добавить cron-job или boot-cleanup: `DELETE FROM program_queue_log WHERE created_at < date('now', '-90 days')`. Аналогично для `float_events`.

### З3. Нет метрики «средняя задержка в очереди» для диагностики

**Секция спеки:** 7.3

`program_queue_log` имеет `wait_seconds`, но нет агрегации. Для пользователя полезно видеть: «за последнюю неделю средняя задержка в очереди группы 1 = 12 мин». Это подсказка — нужно пересмотреть расписание.

**Решение:** Добавить endpoint `GET /api/queue/stats` с агрегацией по группам.

---

### З4. SSE event naming — нет namespace

**Секция спеки:** 6.1

Новые SSE events: `queue_update`, `float_pause`, `float_resume`. Рекомендуется namespace: `queue:update`, `float:pause`, `float:resume` — для консистентности и возможности фильтрации.

---

### З5. float_timeout_minutes=30 — может быть мало для глубоких скважин

**Секция спеки:** 3.2

30 минут — разумно для ёмкости с поплавком. Но если ёмкость большая и насос подкачки медленный, восстановление может занять 40-60 минут.

**Решение:** Значение по умолчанию 30 мин OK, но в UI подсказать: «Установите таймаут с учётом времени заполнения вашей ёмкости. Если ёмкость не восстановится за это время — аварийный стоп.»

---

### З6. Нет dry-run режима для тестирования очереди

Для отладки полезен режим, когда очередь работает, но зоны реально не включаются. Можно добавить позже.

---

### З7. Worker thread naming

`Thread(target=_worker, args=(gid,))` — нет имени потока. При 5 группах в `top -H` видно 5 безымянных Thread. 

**Решение:** `Thread(target=_worker, args=(gid,), name=f"queue-worker-{gid}", daemon=True)`.

---

### З8. Нет механизма ручного «протолкнуть» entry в начало очереди

Если пользователь видит, что его программа третья в очереди — нет способа поднять приоритет. Спека явно отвергает приоритеты (FIFO), но ручная «перестановка» через UI могла бы быть полезна.

---

## 4. ПОДТВЕРЖДЕНО (что в спеке сделано правильно)

### П1. Per-group queue вместо глобальной — ✅ Отлично

Архитектурно верное решение. Разные насосы = разные группы = параллельность. Соответствует подходу OpenSprinkler Sequential Groups. Глобальная очередь (как в Hunter) убила бы параллельность.

### П2. FIFO без приоритетов — ✅ Правильно

Приоритеты добавили бы complexity без реальной пользы. Priority inversion, starvation — зачем это в системе полива? FIFO предсказуемо и просто.

### П3. Разбивка программы на сегменты по группам — ✅ Элегантно

Программа с зонами из разных групп автоматически распараллеливается по группам. Это решает проблему из анализа (секция 3.4 overlap-analysis), где `_run_program_threaded()` обрабатывает зоны последовательно даже из разных групп.

### П4. Двухуровневая защита (FloatMonitor + wb-rules watchdog) — ✅ Критически важно

wb-rules на контроллере — последний рубеж. Если Python-сервис завис — wb-rules выключит реле за миллисекунды. Ни один коммерческий контроллер не имеет такой архитектуры. Уникальное преимущество.

### П5. Дебаунс поплавка 5 сек — ✅ Разумно

Защита от электрических помех и кратковременных колебаний уровня. 5 секунд — стандарт индустрии для дискретных датчиков.

### П6. max_wait_time 120 мин — ✅ Практично

Поливать в 08:00+ если программа была на 06:00 — уже плохо (солнце, испарение). 120 минут — достаточный буфер для разумных пересечений, но не бесконечное ожидание.

### П7. Weather check на уровне программы (skip) + на уровне зоны (adjust) — ✅ Правильная архитектура

Skip целиком перед enqueue — не засоряем очередь. Adjust per zone — гранулярность. Двухуровневый подход.

### П8. check_program_conflicts с WARNING/ERROR уровнями — ✅ Хороший UX

Не блокирует сохранение — очередь гарантирует безопасность в рантайме. Но предупреждает пользователя — помогает оптимизировать расписание.

### П9. Порядок реализации (8 этапов) — ✅ Грамотная декомпозиция

Этапы 1+2+3 = MVP. Поплавок, UI, API — позже. Правильный инкрементальный подход.

### П10. Тест-сценарии T1-T18 — ✅ Покрытие ключевых кейсов

Хорошее покрытие happy path и основных edge cases. Но нужно добавить тесты на проблемы из этого ревью (см. К1-К6).
